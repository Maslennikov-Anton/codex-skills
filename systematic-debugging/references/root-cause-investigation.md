# Root cause investigation

## Что собирать до исправления

Минимальный набор:

- точные шаги воспроизведения;
- тексты ошибок, stack trace, коды ответов;
- временные метки и окружение;
- последние изменения в коде, конфиге, зависимостях или данных.

Если воспроизвести проблему нельзя стабильно, не придумывай fix по интуиции. Сначала увеличь объем наблюдений.

## Multi-component systems

Если система состоит из нескольких слоев, например UI -> API -> service -> DB или CI -> build -> deploy -> runtime, добавляй диагностику на каждую границу:

- что входит в компонент;
- что выходит из компонента;
- какие env/config реально доступны;
- где впервые появляется неверное состояние.

Это помогает найти не "место ошибки", а место потери контракта.

## Проверка гипотез

Правильный порядок:

1. Сформулируй одну конкретную гипотезу:
   `"Причина в X, потому что наблюдаем Y и Z"`.
2. Выбери минимальную проверку:
   - дополнительный лог;
   - точечный локальный repro;
   - один маленький эксперимент;
   - один targeted test.
3. Проверь гипотезу.
4. Если не подтвердилась, не наращивай старый fix, а формулируй новую гипотезу.

## Когда надо остановиться и пересмотреть подход

Сигналы:

- уже было 3 и более неудачных fix-попыток;
- каждый новый fix ломает соседнюю часть системы;
- выясняется скрытое общее состояние или сильная связность;
- "простая правка" превращается в цепочку заплаток.

В этом случае проблема может быть архитектурной, а не локально-имплементационной.

## Practical checklist

- Проблема воспроизводится или ее невоспроизводимость описана.
- Есть evidence по границам компонентов.
- Есть одна основная гипотеза.
- Есть минимальная проверка гипотезы.
- Fix обсуждается только после подтверждения причины.
