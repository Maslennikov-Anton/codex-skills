# Modbus в VCont

## Встроенный Modbus Server

VCont runtime умеет запускать встроенный Modbus TCP server из `vcont.fboot`. Сервер создается как функциональный блок типа `MBSERVER`, параметры передаются через `Options`, затем сервер запускается отдельной командой `START`.

```xml
;<Request ID="2" Action="CREATE"><FB Name="MODBUS_1" Type="MBSERVER" Options="Address=0.0.0.0 Port=1502 Mode=tcp SlaveId=1"/></Request>
MODBUS_1;<Request ID="4" Action="START"/>
```

Ключевые параметры:

- `Address`: адрес, на который привязывается сервер, например `0.0.0.0`.
- `Port`: TCP-порт, например `1502`.
- `Mode`: `tcp` для клиентов `MBCLIENTTCP`, `rtu` для клиентов `MBCLIENTRTUOVERTCP`.
- `SlaveId`: идентификатор slave-устройства.

В сохраненных требованиях указано, что разделитель параметров - пробел. В реальных `.fboot` встречается и позиционная часть `Options="0.0.0.0 1502 Address=0.0.0.0 Port=1502 Mode=tcp SlaveId=1"`; при разборе конкретного bootfile ориентируйся на фактически поддерживаемый runtime.

## Alias-связи

Alias похож на `Connection`, но связывает ножку функционального блока с регистром Modbus server. Alias однонаправленный.

ФБ -> сервер:

```xml
TASK_1;<Request ID="6" Action="CREATE"><Alias Source="PICALC.PI" Destination="MODBUS_1.MD10" /></Request>
```

Сервер -> ФБ:

```xml
TASK_1;<Request ID="6" Action="CREATE"><Alias Source="MODBUS_1.ML1" Destination="PICALC.N" /></Request>
```

Смысл направлений:

- `Source="PICALC.PI" Destination="MODBUS_1.MD10"`: выходная ножка `PI` блока `PICALC` публикуется в регистр `MD10`.
- `Source="MODBUS_1.ML1" Destination="PICALC.N"`: регистр `ML1` сервера записывается во входную ножку `N` блока `PICALC`.

Текущая реализация alias не использует универсальную внешнюю шину runtime-server. Вместо нее:

- для исходящих alias у ФБ есть таблица по индексам выходных ножек;
- у Modbus server есть hash table изменений alias;
- изменения выходов ФБ отслеживаются в `CFunctionBlock`, после чего server получает уведомление и обновляет mapping;
- уведомления об обновлении реализованы через localhost UDP socket, по сути как локальный pub-sub;
- для входящих alias у Modbus server есть статическая таблица регистров, где элемент содержит вектор пар `ФБ - номер входной ножки`, что дает мультиaliasing;
- при записи внешнего Modbus-клиента server обновляет входящие alias-таблицы соответствующих ФБ, а ФБ затем переносит новые значения во входные ножки.

## Имена регистров alias

Упрощенный формат адреса: `<Код><Адрес>`.

Коды:

- `QX`: coil, 1 бит, RW, адреса `QX0-QX65535`.
- `IX`: discrete input, 1 бит, RO, адреса `IX0-IX65535`.
- `IW`: input register, 16 бит, RO, адреса `IW0-IW65535`.
- `MW`: holding register, 16 бит, RW, alias `MW0-MW29999`, Modbus-адреса `0-29999`.
- `MD`: holding register, 32 бит, RW, alias `MD0-MD9999`, Modbus-адреса `30000-49998`.
- `ML`: holding register, 64 бит, RW, alias `ML0-ML3750`, Modbus-адреса `50000-65000`.

Адреса начинаются с нуля: `QX3` означает coil по адресу 3, четвертый по счету.

## Асинхронный Modbus-клиент

Асинхронный обмен разработан для коммуникационного слоя `modbusbatch`, связанного с ФБ `MBREAD` и `MBWRITE`, без изменения `libmodbus`. Идея - отправить пачку запросов сразу, затем принять и сопоставить ответы в пределах общего timeout. Это убирает накопление timeout по каждому диапазону или регистру.

В runtime сохранены оба режима:

- `modbusbatch`: синхронный режим.
- `modbus_async`: асинхронный режим.

Асинхронный режим особенно полезен при большом числе непоследовательных регистров и заметной сетевой задержке. В ранних испытаниях чтение 1000 регистров при сетевой задержке около 100 мс занимало примерно `105.48` с синхронно и `0.33` с асинхронно.

Ограничения и риски:

- Async-клиент не создает поток на каждый регистр; используется раздельная отправка и прием больших TCP-фрагментов с Modbus-запросами/ответами.
- Реальная пропускная способность зависит от Modbus server/PLC: некоторые устройства не успевают обработать большую пачку запросов.
- Для проблемных устройств можно ограничивать число запросов за один раз.
- Например, Tesys T обрабатывал не больше 4 async-запросов в части диапазонов и возвращал `Server busy` при превышении.

## MBREAD

`MBREAD` поддерживает синхронный и асинхронный режимы. Режим задается строкой подключения.

Пример ID без ограничения пакетов:

```text
modbus_async[127.0.0.1:1502:1]
```

Пример ID с ограничением пакетов:

```text
modbus_async[127.0.0.1:1502:1:4]
```

Параметры:

- `127.0.0.1`: адрес Modbus server.
- `1502`: порт.
- `1`: slave ID.
- `4`: опциональное ограничение количества пакетов за один раз; если не указано, по умолчанию `0`, то есть без ограничения.

Входы `RA1...RA100` в async-режиме задаются строками:

```text
"REG9:1000:3:79"
```

Поля:

- `REG9`: имя регистра.
- `1000`: период опроса в миллисекундах.
- `3`: Modbus-функция; поддерживаются функции чтения `1`, `2`, `3`, `4`.
- `79`: адрес регистра.

Важное ограничение: в async-режиме `MBREAD` принимает строго один адрес на один вход. Диапазон адресов на одном входе недопустим; клиент спроектирован под однорегистровые Modbus-запросы.

Ошибки:

- Если адрес прочитать не удалось, соответствующий выход данных, например `RD01`, получает `-1`.
- Выход качества, например `RQ01`, содержит код Modbus-ошибки, увеличенный на 1. Чтобы получить исходный exception code, нужно вычесть 1.
- При потере сетевого соединения значения `RD01...RD100` сохраняются такими, какими были до потери связи.

Modbus exception codes, зафиксированные для async `MBREAD`:

- `1`: Illegal Function.
- `2`: Illegal Address.
- `3`: Illegal Value.
- `4`: Server Failure.
- `5`: Acknowledge.
- `6`: Server Busy.
- `7`: Negative Acknowledge.
- `8`: Memory Parity Problem.
- `10`: Gateway Path Unavailable.
- `11`: Gateway Target Device Failed to Respond.
- `UNKNOWN`: отдельный 11-й внутренний вариант вне перечня exception codes.

## MBWRITE

`MBWRITE` поддерживает синхронный и асинхронный режимы. Async-режим работает с Modbus-функциями:

- `5`: Write Single Coil.
- `6`: Write Single Register.

Строка ID допускает 3, 4 или 5 параметров в квадратных скобках. Первые 3 обязательны, 4-й и 5-й опциональны:

```text
modbus_async[127.0.0.1:1502:1]
modbus_async[127.0.0.1:1502:1:2]
modbus_async[127.0.0.1:1502:1:0:confirm]
```

Параметры:

- `ip`: адрес Modbus server, куда выполняется запись.
- `port`: порт, допустимы значения от `502` и выше.
- `slave id`: slave ID.
- `limit`: опциональное ограничение количества пакетов за один раз, по умолчанию `0`, то есть без ограничения.
- `confirm`: опциональное ключевое слово, включает ожидание подтверждения записи и влияет на качество выходов `WQ1...WQ100`.

Если строка ID некорректна, runtime принимает значения по умолчанию:

- `127.0.0.1`
- `1502`
- `1`
- `0`
- `false` для `confirm`

Входы `WA1...WA100` задаются строками:

```text
"REG10:6:10"
```

Поля:

- `REG10`: имя регистра.
- `6`: Modbus-функция записи.
- `10`: адрес регистра.

Данные для записи подаются на `WD1...WD100` связями от выходов других блоков. Для функции `6` поддерживаются `INT`/`UINT` длиной до 16 бит; для функции `5` - до 8 бит.

Поведение `confirm`:

- если `confirm` указан, `MBWRITE` после отправки пачки запросов ждет ответы до timeout и выставляет качество по фактическим подтверждениям;
- если `confirm` не указан, ожидание ответов не выполняется, а внутренний флаг подтверждения равен `false`;
- чтобы включить `confirm`, его нужно указывать именно пятым параметром, то есть вместе с четвертым параметром `limit`.
