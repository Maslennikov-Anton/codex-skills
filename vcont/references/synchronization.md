# Синхронизация и канал связи

## Канал контроллеров

Для синхронизации данных нужен один или несколько специально выделенных сетевых интерфейсов виртуального контроллера. Эти интерфейсы не используются для обмена с IDE или с полевыми контроллерами.

Модель обмена - client/server, но поскольку кластер может содержать больше двух виртуальных контроллеров и внешний арбитр/voter не предполагается, каждый контроллер может работать и клиентом, и сервером. Архитектура является peer-to-peer поверх TCP.

## Установление соединения

У каждого виртуального контроллера есть таблица IP-адресов всех соседних виртуальных контроллеров.

При старте контроллер:

1. Запускает сервер на заранее согласованном порту кластера.
2. Для каждой записи в таблице peers пытается подключиться как клиент.
3. Если клиентское соединение не удалось, повторяет попытку через настроенный таймаут.
4. Если удаленный контроллер подключился первым во время таймаута, принимает это соединение и считает пару локальный-удаленный контроллер подключенной.
5. Если до истечения таймаута соединение не установлено, повторяет алгоритм с шага повторной попытки.

При разрыве соединения процедура установления соединения запускается заново без перезапуска сервера. Каждый контроллер держит пул соединений со всеми peers и отслеживает их актуальные статусы.

## Модель синхронизации

Контроллеры в режиме синхронизации образуют кластер. Один контроллер является Master/основным; остальные - Slave/резервными. В любой момент времени только один контроллер может быть Master.

Синхронизация реализуется внутри исполняемого файла виртуального контроллера и работает в контексте процесса VCont. Она включается на уровне всего контроллера (`CDevice`), там же задается роль синхронизации.

## Критерии синхронизации

Синхронизация происходит каждый раз, когда меняются данные. Документация выделяет два сценария изменений:

- Пользователь online меняет входные значения функциональных блоков, например адрес регистра у `MBREAD`.
- Функциональный блок завершил работу и его выходные данные изменились, поэтому следующий ФБ будет работать с новыми данными.

Оба сценария сводятся к изменению входных или выходных данных функционального блока. Поэтому единица синхронизации - функциональный блок (`CFunctionBlock`).

Каждый функциональный блок должен знать свою роль в синхронизации через владеющий им экземпляр `CDevice` и участвовать в синхронизации согласно этой роли.

Важное сохраненное ограничение: синхронизация только входов и выходов ФБ предполагает, что ФБ stateless. Если у ФБ есть промежуточное состояние между выполнениями, синхронизации только входов/выходов может быть недостаточно.

## Поведение Master

Основной виртуальный контроллер действует как Master и выполняет фактическую работу по автоматизированному управлению технологическим процессом.

По заранее установленным каналам связи он уведомляет все соседние/резервные контроллеры об изменениях при обработке каждого ФБ. В первой версии можно отправлять все входные и выходные данные ФБ; последующая оптимизация может отправлять только измененные данные, по аналогии с rsync.

## Поведение Slave

Резервный виртуальный контроллер действует как Slave. Он не выполняет работу по управлению технологическим процессом, несмотря на то что в него загружено то же управляющее приложение.

Его цель - поддерживать входы и выходы всех ФБ в актуальном состоянии, как если бы он активно выполнялся. При получении изменения состояния ФБ от основного контроллера он обновляет входные/выходные параметры ФБ и выставляет маркер выполнения для этого ФБ в рамках одного таска.

## Docker и eCAL

При запуске VCont/eCAL в Docker могут потребоваться дополнительные настройки eCAL и сетевого окружения контейнеров. Проверяй официальную документацию eCAL по Docker: https://eclipse-ecal.github.io/ecal/v5.13/getting_started/cloud.html

Ключевые моменты из документации eCAL:

- Для сетевого обмена eCAL внутри контейнера может потребоваться включить `network_enabled`.
- Для локальной shared-memory коммуникации между host и контейнерами eCAL рекомендует запуск с общими namespace: `--ipc=host`, `--pid=host`, иногда `--network=host`.
- Если контейнеры должны обмениваться eCAL-данными с host или внешней сетью через bridge-сеть, может потребоваться multicast route на host, например маршрут для группы `239.0.0.0/24` на docker bridge-интерфейс.
- Для IPC-коммуникации через границы host/container в eCAL 5.13 важен общий `shm_transport_domain` в `ecal.yaml`.
- В этом проекте используется `ecal.ini`, а не `ecal.yaml`; поэтому применяй идеи из eCAL Docker-документации с учетом фактического формата конфигурации поставки VCont.
