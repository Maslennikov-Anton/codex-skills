# Жизненный цикл технологий и подходов

## Статусы

### experimental

Использовать точечно, чтобы получить практический опыт.

Критерии:
- технология новая для команды;
- есть польза, но нет подтвержденной устойчивости;
- разрешена только в ограниченных задачах.

### preferred

Использовать по умолчанию для новых задач этого класса.

Критерии:
- минимум два успешных применения;
- понятные плюсы по скорости, качеству или поддерживаемости;
- нет заметных системных минусов.

### legacy-compatible

Поддерживать в существующем коде, но не продвигать в новые решения без причины.

Критерии:
- технология еще жива в текущих проектах;
- для новых задач уже есть более сильный стандарт.

### deprecated

Не выбирать для новых решений.

Критерии:
- есть более сильная альтернатива;
- стоимость поддержки выше пользы;
- технология нужна только из-за исторического хвоста.

## Вопросы перед сменой статуса

1. Есть ли подтвержденный практический опыт?
2. Снижает ли это решение число повторных архитектурных споров?
3. Улучшает ли оно скорость или качество?
4. Не создаст ли новый стандарт лишнюю сложность для большинства задач?

## Примеры для текущей среды

- `SQLAlchemy + Pydantic` для Python SQL/data-access -> `preferred`
- новый экспериментальный ORM или framework -> сначала `experimental`
- старый точечный подход, который еще живет в наследуемом коде -> `legacy-compatible`
