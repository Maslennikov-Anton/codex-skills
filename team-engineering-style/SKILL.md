# Локальный инженерный стиль

Этот skill задает внутренние правила команды поверх базовых профессий. Он нужен не для замены `python-developer`, `autotest-engineer`, `analyst` и других skills, а для фиксации локальных стандартов, которые должны последовательно применяться в работе.

## Когда использовать

Используй этот skill, если нужно:

- зафиксировать устойчивое правило после выполненной задачи;
- обновить предпочтительный стек, подход или архитектурный паттерн;
- решить, является технология экспериментальной, предпочтительной или legacy-compatible;
- убрать противоречия между локальными skills;
- выделить общий стандарт, который относится сразу к нескольким ролям;
- договориться, как skills должны эволюционировать со временем.

## Границы ответственности

- Базовые skills отвечают за профессию и предметную область.
- Этот skill отвечает за локальные стандарты команды и правила их развития.
- Изменения в этот skill должны быть короче и стабильнее, чем изменения в проектном коде.
- Если правило относится только к одной профессии, сначала обновляй самый узкий skill, а этот skill меняй только если правило стало кросс-функциональным.

## Текущие локальные правила по умолчанию

- Для Python-кода использовать явную типизацию аргументов, возвращаемых значений, переменных и промежуточных коллекций там, где это улучшает читаемость и не перегружает код.
- Для SQL/data-access по умолчанию предпочитать `SQLAlchemy + Pydantic`, если в репозитории уже не закреплен другой стек, который нужно сохранять.
- Для структурных комментариев писать короткие технические комментарии на русском языке.
- Все новые skills и обновления существующих skills по умолчанию писать на русском языке: `SKILL.md`, `references/`, комментарии в bundled resources и пользовательские поля в `agents/openai.yaml` должны быть на русском, если нет явно зафиксированного исключения.
- Для API- и интеграционных автотестов по умолчанию использовать `pytest + httpx + allure`, если ограничения проекта не требуют другого.
- После реализации feature, bugfix, refactor, DB change или API change по умолчанию выполнять минимум `lint + tests + review` перед завершением задачи.
- Если изменение затронуло запуск проекта, стек, quality gates, структуру модулей, публичные сценарии работы или другие вещи, важные для следующего инженера, нужно актуализировать `README` и связанную документацию в рамках той же задачи.
- Падающий тест по умолчанию трактовать как сигнал о потенциальном баге продукта, регрессии или нарушении контракта, а не как повод автоматически подгонять тест под текущее поведение.
- Для тестовых и контрактных решений использовать такой порядок выбора источника истины:
- если есть явная спецификация, acceptance criteria или продуктовый контракт, они являются эталоном;
- если спецификации нет или она не покрывает часть поведения, решение принимать по здравому смыслу, пользовательскому опыту и общему направлению продукта;
- фактическое поведение рантайма использовать как evidence для диагностики, но не как автоматическую замену спецификации.
- Для подтвержденных product/backend defects в тестовом проекте использовать `xfail` только как управляемый механизм фиксации известной проблемы, а не как способ скрыть непонятное падение.
- Если проект поддерживает несколько режимов запуска, например host-run и compose-run, defaults в коде, `.env.example`, `README` и CI не должны расходиться.
- Для тестовых проектов с API- или интеграционным покрытием изменения в coverage, `xfail`, known bugs и test contract должны сопровождаться обновлением документированных артефактов: coverage matrix, bug artifact или эквивалентной документации.
- Для новых локальных правил сначала выбирать наиболее узкий уровень фиксации: конкретный skill, затем кросс-рольный skill, затем system-skill только при реальной необходимости.
- Если добавлен новый skill или изменен существующий skill, изменения нужно закоммитить и запушить в GitHub-репозиторий `codex-skills` в рамках той же задачи, если пользователь явно не попросил этого не делать.
- Если задача распараллелена на нескольких sub-agent'ов, финальный разбор результатов начинать только после завершения всех запущенных агентов. Не отвечать пользователю по неполному набору агентских результатов, если только пользователь явно не попросил промежуточный отчет.
- После завершения задачи sub-agent'а его тред нужно закрывать, если он больше не нужен, чтобы не копить висящие агентские сессии и не упираться в лимиты инструментов.

## Модель внедрения новых технологий

Каждая новая технология, библиотека, фреймворк или подход должны иметь один из статусов:

1. `experimental` — пробуем точечно, собираем опыт, не делаем стеком по умолчанию.
2. `preferred` — используем по умолчанию для новых задач этого класса.
3. `legacy-compatible` — поддерживаем в существующих проектах, но не продвигаем в новые решения без причины.
4. `deprecated` — не выбираем для новых решений, допускаем только в режиме поддержки старого кода.

## Правила перевода технологии между статусами

- `experimental` -> `preferred`, если подход успешно применился минимум в двух реальных задачах и не создал заметного операционного или архитектурного шума.
- `preferred` -> `legacy-compatible`, если команда перестала выбирать этот подход по умолчанию, но в активных проектах он еще нужен.
- `legacy-compatible` -> `deprecated`, если поддержка остается только из-за исторического хвоста и есть лучший стандарт.
- Не повышай статус технологии только из-за интереса к новинке; нужен хотя бы короткий практический опыт.

## Workflow обновления skills

1. Зафиксируй триггер для изменения:
- повторяющееся решение;
- новый устойчивый стек;
- найденное противоречие;
- устаревшее правило;
- успешный или неудачный опыт, который имеет смысл сохранить.

2. Определи уровень изменения:
- только один skill;
- несколько профессий;
- общий локальный стандарт;
- system-level правило.

3. Выбери тип изменения:
- `workflow` — как работать;
- `stack` — что использовать;
- `quality rule` — как проверять качество;
- `boundary` — где skill должен и не должен применяться.

4. Обнови самый узкий подходящий skill.

5. Проверь согласованность:
- нет ли противоречий с соседними skills;
- не дублируется ли правило;
- не стало ли описание слишком общим или слишком хрупким.

6. После правки проверь:
- валидность skill;
- наличие битых ссылок на references и scripts;
- понятность trigger в `description`.
7. Если был создан новый skill или изменен существующий, зафиксируй изменения в git-репозитории skills и запушь их в GitHub, если пользователь явно не запретил push.

## Правила качества для skill-эволюции

- Не раздувай skill ради единичного исключения.
- Не повышай новую практику до правила по умолчанию без фактического опыта.
- Не смешивай в одном skill общую профессию и локальную политику команды, если можно разделить.
- Любое новое правило должно отвечать хотя бы на один из вопросов:
- что ускоряет работу;
- что снижает количество ошибок;
- что уменьшает число повторных решений;
- что упрощает выбор между альтернативами.
- Если правило нельзя объяснить одной-двумя короткими причинами, скорее всего оно еще не дозрело до skill.

## Локальный стандарт для тестового контракта

Для API- и интеграционных тестовых проектов по умолчанию поддерживай такой набор артефактов:

- тестовый код;
- актуальный `README` с описанием запуска и coverage;
- coverage matrix по endpoint'ам и user stories, если проект используется как самостоятельный test asset;
- отдельный bug artifact или эквивалентный документ для известных `xfail`-дефектов.

Если контракт тестов массово пересматривается, сначала собирай evidence из рантайма, а потом меняй ожидания пакетно.

## Рекомендуемый операционный ритм

- Сразу после крупной задачи: короткая правка самого узкого релевантного skill.
- Раз в 1-2 недели: ревью локальных skills на противоречия, дубли и устаревание.
- После появления нового рабочего стека: сначала `experimental`, затем повторная оценка после накопления опыта.

## Стандарт завершения реализации

По умолчанию завершение реализации считается корректным только если:

1. Код прошел автоматические проверки, доступные в проекте.
2. Прогнаны релевантные тесты.
3. Изменения прошли профессиональное ревью.

Если проект поддерживает formatter, linter, type checks и тесты, стандартный порядок такой:

1. formatter;
2. linter;
3. type checks;
4. tests;
5. review.

Для низкорисковых правок без изменения поведения допускается облегченный режим. Для feature, bugfix, refactor, DB change, API change и инфраструктурных изменений использовать полный порядок по умолчанию.

## Уровни строгости проверки

| Уровень риска | Типичные примеры | Обязательный минимум |
| --- | --- | --- |
| `low` | правка текста, безопасный rename, локальный комментарий без изменения поведения | базовая самопроверка, formatter/linter при наличии |
| `medium` | обычная feature, bugfix, локальный refactor, изменение тестового слоя | formatter, linter, type checks при наличии, релевантные тесты, review |
| `high` | DB change, API change, авторизация, инфраструктура, кросс-модульный refactor | formatter, linter, type checks, релевантные и расширенные тесты, review с акцентом на регрессии и риски |

## Карта reference-файлов

Читай только то, что нужно:

- `references/evolution-process.md` -> детальный порядок развития skills и принятия новых правил.
- `references/technology-lifecycle.md` -> жизненный цикл технологий и критерии перехода между статусами.
- `references/implementation-done-definition.md` -> критерии завершения реализации и стандарт post-implementation проверки.
- `references/role-selection-matrix.md` -> быстрый выбор роли или связки ролей под конкретный тип задачи.

## Формат ответа

Когда просят обновить или развить локальные стандарты, возвращай:

1. Что именно меняется и почему.
2. На каком уровне нужно закрепить правило.
3. Какой статус получает технология или подход.
4. Какие skills нужно обновить.
5. Как проверить, что новое правило реально улучшило работу.
