# Локальный инженерный стиль

Этот skill задает общие локальные правила команды поверх базовых профессий. Он нужен для кросс-рольных стандартов и эволюции самих skills, а не для хранения подробных инструкций по конкретной профессии.

## Когда использовать

Используй этот skill, если нужно:

- зафиксировать новое устойчивое правило команды;
- определить статус технологии или подхода;
- убрать противоречие между несколькими локальными skills;
- понять, на каком уровне нужно закрепить правило;
- обновить процесс эволюции skills.

## Границы ответственности

- Профильные skills отвечают за профессию и предметную область.
- Этот skill отвечает только за кросс-рольные стандарты и правила развития локальной базы знаний.
- Если правило относится к одной профессии, сначала обновляй самый узкий skill.
- Если правило оказалось одноразовым workaround'ом или не переживает больше одной задачи, не поднимай его в локальный стандарт.

## Текущие локальные правила по умолчанию

- Новые skills и обновления существующих skills по умолчанию писать на русском языке: `SKILL.md`, `references/`, bundled resources и пользовательские поля в `agents/openai.yaml`, если нет явно зафиксированного исключения.
- После изменения или добавления локального skill изменения нужно закоммитить и запушить в репозиторий `codex-skills` в рамках той же задачи, если пользователь явно не попросил этого не делать.
- Если задача распараллелена на нескольких sub-agent'ов, финальный разбор результатов начинать только после завершения всех запущенных агентов, если пользователь явно не попросил промежуточный отчет.
- После завершения работы sub-agent'а его тред нужно закрывать, если он больше не нужен.

## Модель статусов технологий

Каждая новая технология, библиотека, фреймворк или подход должны иметь один из статусов:

1. `experimental` — пробуем точечно, собираем опыт, не делаем стеком по умолчанию.
2. `preferred` — используем по умолчанию для новых задач этого класса.
3. `legacy-compatible` — поддерживаем в существующих проектах, но не продвигаем в новые решения без причины.
4. `deprecated` — не выбираем для новых решений, допускаем только в режиме поддержки старого кода.

Не повышай статус технологии без реального повторяемого опыта.

## Workflow обновления skills

1. Зафиксируй триггер изменения: повторяющееся решение, найденное противоречие, устаревшее правило, новый устойчивый стек.
2. Определи самый узкий подходящий уровень фиксации: конкретный skill, кросс-рольный skill, system-level правило.
3. Обнови минимально необходимый набор skills.
4. Проверь, что новое правило не дублирует уже существующее и не тянет одноразовую проектную специфику.
5. Проверь валидность skill, ссылки на `references/` и понятность `description`.
6. Зафиксируй изменение в git, если пользователь это не запретил.

## Правила качества для skill-эволюции

- Не раздувай основной `SKILL.md` подробными manual-разделами, если их можно вынести в `references/`.
- Не смешивай в одном skill профессию, локальную политику команды и проектно-специфичный workaround.
- Любое новое правило должно ускорять работу, снижать число ошибок или устранять повторный выбор.
- Если правило нельзя объяснить одной-двумя короткими причинами, скорее всего оно еще не дозрело до локального стандарта.

## Карта reference-файлов

Читай только то, что нужно:

- `references/evolution-process.md` -> детальный порядок развития skills и принятия новых правил.
- `references/technology-lifecycle.md` -> жизненный цикл технологий и критерии перехода между статусами.
- `references/implementation-done-definition.md` -> критерии завершения реализации и стандарт post-implementation проверки.
- `references/role-selection-matrix.md` -> быстрый выбор роли или связки ролей под конкретный тип задачи.

## Формат ответа

Когда просят обновить или развить локальные стандарты, возвращай:

1. Что именно меняется и почему.
2. На каком уровне нужно закрепить правило.
3. Какие skills нужно обновить.
4. Как проверить, что новое правило реально улучшило работу.
