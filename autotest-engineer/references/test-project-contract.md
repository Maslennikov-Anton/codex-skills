# Test Project Contract

Используй этот reference, когда тестовый репозиторий сам является поддерживаемым инженерным продуктом.

## Что должно быть согласовано

- coverage matrix по endpoint-ам, user stories или ключевым сценариям;
- отдельный bug registry или другой bug artifact, который не совпадает с самими test cases;
- `README`, `.env.example`, runtime defaults и CI jobs;
- ignore-правила для generated artifacts.

## Правила поддержки

- Разделяй сущности `regression case`, `bug reproducer` и `bug registry`; не смешивай статус дефекта с самим тестовым кейсом.
- Разделяй статусы `covered`, `covered with known bug`, `uncovered`.
- Считай красный тест допустимым и ожидаемым итогом, если он честно обнаружил реальный дефект или сломанный контракт.
- Не выравнивай pass-rate за счет потери проверочной силы: слабых assertion-ов, чрезмерных mock-ов, условных skip-ов и silent fallback-ов.
- Не храни `allure-results`, временные отчеты, `junit`, `*.egg-info`, локальные кэши и похожие generated-файлы в git.
- Каждый тестовый кейс должен проходить в своей ожидаемой категории: positive кейс через success oracle, negative кейс через expected-failure oracle.
- `xfail`, `skip` и quarantine не должны быть хранилищем правды о багах. Их нельзя использовать как долгоживущий inventory известных дефектов.
- Если defect reproducer перестал воспроизводиться, его нужно немедленно переводить в stale/fixed candidate и выводить из `xfail`-контура.
- Для подтвержденных дефектов `xfail` допустим только как кратковременный переходный механизм с конкретным `reason`, bug id, датой последней проверки и явной реверификацией.
- `xfail`, `skip` и quarantine допустимы только как явно задокументированное исключение, а не как способ скрыть неизвестную, неудобную или уже потенциально закрытую поломку.
- Перед массовым изменением тестов сначала собери evidence: фактические ответы, состояние данных и воспроизводимый сценарий.
- Если тест перестал ловить дефект после рефакторинга, это регрессия тестового набора, даже если CI снова стал зеленым.
